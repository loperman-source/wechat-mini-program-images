一、产品概述
项目名称：来人上菜
核心目标：基于用户选择的餐别（正餐、甜品、夜宵），通过单次点击智能推荐匹配菜品，为用户提供快速、有趣的用餐决策工具。
主要用户：有选择困难、寻求快速用餐建议的年轻用户。

二、核心页面：首页（index）
本小程序只需一个页面，承载所有功能。

2.1 页面布局与状态
页面分为三个核心区域，包含三种交互状态：
flowchart LR
    subgraph A [状态切换逻辑]
        S1[“状态1: 初始/选择状态”]
        S2[“状态2: 匹配中 (加载)”]
        S3[“状态3: 匹配成功 (结果)”]
    end

    S1 -- 用户点击 GO --> S2
    S2 -- 执行匹配逻辑 --> S3
    S3 -- 用户点击“再换一换” --> S2



2.2 各状态详细设计
状态一：初始 / 选择状态

顶部标题区：固定显示“来人上菜”及评分“4.8”。

餐别选择区：

横向排列三个选项卡片：正餐、甜品、夜宵。

当前选中项有视觉高亮（如边框加粗、背景色变化）。

交互：点击任一选项立即选中，并取消其他选项。

中央操作区：

显示一个醒目的大型“GO”按钮。

交互：未选择餐别时按钮为禁用态（灰色）；选择后变为可点击态（主题色）。

底部说明区：固定显示一行提示语，如“点击GO，开启你的美味邂逅！”。

状态二：匹配中（加载）状态

视觉反馈：点击“GO”后，整个中央区域切换为加载动画。

动画示意：可设计为盘子旋转、厨师烹饪等趣味动态效果，并配文“正在为您精心搭配...”。

状态三：匹配成功（结果）状态

结果展示区：

菜品图片：高清、诱人的菜品照片，占显眼位置。

菜品名称：大号字体显示，如“麻辣香锅”。

菜品分类标签：紧随名称后，如“川湘菜”。

菜品描述：显示描述文案，如“香辣过瘾，食材丰富...”。

特色标签：以标签形式展示2-3个关键词，如“下饭神器”、“重口味”。

底部操作区：

显示一个“再换一换”按钮。

交互：点击后，基于当前已选餐别重新匹配，页面切回“状态二”再进入“状态三”。

三、核心功能流程
用户进入页面：处于“状态一”。

用户选择餐别：点击“正餐”、“甜品”或“夜宵”之一。

用户触发匹配：点击“GO”按钮。

系统执行匹配：

页面进入“状态二”（加载）。

根据所选餐别，从小程序本地的菜品库中随机筛选出一个对应菜品。

模拟网络延迟（建议0.3-0.5秒）后，进入“状态三”。

用户查看结果：展示匹配的菜品详情。

用户重新匹配：点击“再换一换”，重复步骤4-5。

四、数据与逻辑
4.1 本地菜品数据示例
建议将菜品数据以数组形式存储在 index.js 的 data 中或一个独立的 .js 文件里。

4.2 核心匹配逻辑
在 index.js 中实现一个匹配函数：

javascript
// 当用户点击GO或再换一换时触发
matchDish() {
  const selectedType = this.data.selectedType; // 用户当前选择的餐别
  const allDishes = this.data.dishes;
  // 1. 过滤出对应餐别的所有菜品
  const matchedPool = allDishes.filter(dish => dish.type === selectedType);
  // 2. 从池中随机选取一个
  const randomIndex = Math.floor(Math.random() * matchedPool.length);
  const resultDish = matchedPool[randomIndex];
  // 3. 将结果设置到data中，触发页面渲染
  this.setData({
    currentDish: resultDish,
    pageStatus: 'success' // 控制页面状态
  });
}
五、非功能性要求
性能：因无后端交互，匹配响应应瞬时完成（加载动画仅作体验缓冲）。

兼容性：严格遵循微信小程序开发规范，确保在iOS、Android及HarmonyOS上的微信客户端内，样式与交互一致。

UI/UX：

设计风格需与提供的“麻辣香锅”卡片风格一致：简洁、突出美食、标签化。

所有点击操作需有明确的视觉反馈（如按钮按下态）。

图片需预先优化，保证加载速度。
